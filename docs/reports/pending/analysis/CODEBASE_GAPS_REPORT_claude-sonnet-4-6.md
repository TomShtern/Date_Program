> **Generated by:** claude-sonnet-4-6

## Executive Summary

The codebase is architecturally mature: clean domain model, solid storage layer, well-structured async ViewModel infrastructure, and a good separation of concerns. The core matching, filtering, and safety logic is largely production-ready.

The recurring pattern across every layer is **the domain core is rich, but the wiring and exposure layers have the gaps**. Features are often implemented in the core but not surfaced via use-cases; use-case operations exist but are never called from the REST API; UI ViewModels have observable properties that controllers don't bind; events are defined but never published.

This report is organized by impact area.

## 1. Critical Bugs / Runtime Failures

These will cause crashes or silent failures in production.

### 1.1 ~~Missing `daily_picks` Table~~ → Misleading Migration Comment (Corrected)

> **Verified correction:** A second pass confirmed this is NOT a runtime crash. The `daily_picks` table is created by V2 migration (`MigrationRunner.applyV2()`) using `CREATE TABLE IF NOT EXISTS`. Because `MigrationRunner.runAllPending()` always applies V1 then V2 in order, every fresh database gets the table correctly.

However, the comment on `MigrationRunner.java:120-124` is factually wrong:
> *"Fresh databases already have this table from V1 (SchemaInitializer includes it)"*

`SchemaInitializer` (V1) does **not** create `daily_picks`. Only V2 does. The runtime is correct; the comment is not.

**Fix:** Correct the comment on lines 120–124 to say "Created in V2 migration, not in V1 baseline."

### 1.2 `StatsViewModel.fetchMessagesExchanged()` Has No Timeout

The method runs a pagination loop over all conversations to count exchanged messages. There is no upper bound on iterations and no exception handling inside the loop. For users with many conversations, this can hang the async scope indefinitely.

**Fix:** Add a loop limit, or query the count directly from `CommunicationStorage` rather than paginating.

---

### 1.3 `UseCaseUiProfileNoteDataAccess` Crashes Instead of Degrading

When the profile note use-case is unavailable, this adapter wraps its exception in an `IllegalStateException` and re-throws it. Other adapters degrade gracefully. This will crash the ChatViewModel or MatchingViewModel when the use-case is null.

**Fix:** Catch the exception, log a warning, and return an empty result.

## 2. Feature Gaps — Things That Are Missing Entirely

### 2.1 Presence & Typing Indicators (Disabled)

`ChatViewModel` has full property support: `presenceStatusProperty`, `remoteTypingProperty`, and a `refreshPresenceState()` method. `UiPresenceDataAccess` is an interface with a proper contract. But `ViewModelFactory` instantiates only `NoOpUiPresenceDataAccess`, so the feature does nothing in production. The code path is complete but has never been enabled.

**Fix:** Implement a real `UiPresenceDataAccess` that polls or subscribes to presence updates, or remove the dead code if presence is not planned.

---

### 2.2 "GUARDIAN" Achievement Is Unreachable

The `EngagementDomain` defines a `GUARDIAN` achievement (awarded for reporting a fake profile). There is no `Report` event defined in `AppEvent`, no event fired when `SocialUseCases.reportUser()` succeeds, and no handler that checks for this achievement. The achievement can never be unlocked.

**Fix:** Add a `ReportSubmitted` event, fire it from `reportUser()`, and check `GUARDIAN` eligibility in `AchievementEventHandler`.

---

### 2.3 `MatchCreated` Event Is Never Published

`AppEvent.MatchCreated` is defined, but no code path in `MatchingUseCases` or `MatchingService` fires it. The `MetricsEventHandler` and `AchievementEventHandler` both watch for swipe results — `AchievementEventHandler` checks `resultedInMatch == true` on `SwipeRecorded` — so match-based achievements work through a workaround. But nothing downstream reacts to the match as a first-class event (e.g., triggering a notification to both users, or analytics).

**Fix:** Fire `MatchCreated` from `MatchingUseCases.processSwipe()` when a mutual like is detected.

---

### 2.4 No Profile Update REST Endpoint

`ProfileUseCases.saveProfile()` exists and works, but no REST endpoint calls it. There is no `PUT /api/users/{id}/profile` endpoint. The REST API effectively provides read-only access to profile data. The CLI and JavaFX GUI can edit profiles; API clients cannot.

**Fix:** Add a REST endpoint for profile update and wire it to `ProfileUseCases.saveProfile()`.

---

### 2.5 Notifications Not Exposed via REST

`SocialUseCases` has `notifications()` (fetch with read-filter) and `markNotificationRead()`. Neither is exposed as a REST endpoint. Notifications are stored to the database from `NotificationEventHandler` but can only be read via the JavaFX GUI or CLI.

**Fix:** Add `GET /api/users/{id}/notifications` and `POST /api/notifications/{id}/read` endpoints.

---

### 2.6 ~~SafetyController and NotesController Don't Exist~~ → Controllers Exist but Are Thin (Corrected)

> **Verified correction:** Both controllers DO exist. A second pass confirmed the source files are present.

However, they are minimal implementations:
- `SafetyController` has only 3 `@FXML` fields: `rootPane`, `blockedUsersListView`, `emptyStateBox`
- `NotesController` has only 3 `@FXML` fields: `rootPane`, `notesListView`, `emptyStateBox` plus one `@FXML`-annotated private `handleOpenSelected()` suppressed for PMD

Both screens are essentially read-only list views. The Safety screen shows blocked users but provides no in-screen path to block or report someone — users must navigate to the match/chat context to trigger those actions. The Notes screen similarly shows notes but appears to have no inline note creation from this screen (only from `ChatController` and `MatchingController`).

**Fix:** Assess whether the Safety and Notes screens need richer functionality: e.g., an "Unblock" action inline in `SafetyController`, or a "New Note" flow in `NotesController` without requiring an open conversation.

---

### 2.7 Undo Has No REST Endpoint

`MatchingUseCases.undoSwipe()` is fully implemented and backed by `UndoService` with proper atomic deletion and time-window enforcement. No REST endpoint exposes it. API clients cannot undo swipes.

**Fix:** Add `DELETE /api/users/{id}/swipes/latest` or `POST /api/users/{id}/undo`.

---

### 2.8 Standouts Not Exposed via REST

`MatchingUseCases.standouts()` and `markStandoutInteracted()` have no REST counterparts. The standout table, storage, and service are all implemented. Only the JavaFX GUI can use this feature.

**Fix:** Add `GET /api/users/{id}/standouts` and `POST /api/users/{id}/standouts/{standoutId}/interact`.

## 3. Implementation Gaps — Things That Are Half-Done

### 3.1 REST API Bypasses Use-Case Layer in 5 Places

These endpoints call core services directly instead of the use-case layer, skipping event publishing, business validation, and cross-cutting concerns:

| Endpoint | Calls Directly | Should Call |
|----------|---------------|-------------|
| `GET /api/users` | `ProfileService.listUsers()` | `ProfileUseCases` |
| `GET /api/users/{id}` | `ProfileService.getUserById()` | `ProfileUseCases` |
| `GET /api/users/{id}/candidates` | `CandidateFinder` | `MatchingUseCases.browseCandidates()` |
| `GET /api/users/{id}/matches` | `MatchingService` | `MatchingUseCases.listActiveMatches()` |
| `GET /api/conversations/{id}/messages` | `MessagingService.getMessages()` | `MessagingUseCases.loadConversation()` |

The `processSwipe()` vs `recordLike()` inconsistency is especially notable: CLI and JavaFX use the high-level `processSwipe()` (which handles daily pick tracking and events), while the REST API uses the lower-level `recordLike()`.

---

### 3.2 Profile Completion Achievements Never Trigger

`ProfileSaved` is defined in `AppEvent`, but the code that activates a user's profile (the `DRAFT → ACTIVE` transition inside `saveProfile()`) does not fire this event. Achievements `COMPLETE_PACKAGE`, `STORYTELLER`, and `LIFESTYLE_GURU` are all gated on profile-related conditions, but they can only be checked via `AchievementEventHandler`, which only runs when a swipe results in a match. Users who complete their profile but don't swipe won't unlock these achievements.

**Fix:** Fire `ProfileSaved(activated=true)` from `ProfileUseCases.saveProfile()` after a successful activation, and add an `AchievementEventHandler` subscription for it.

---

### 3.3 SELECTIVE and OPEN_MINDED Achievements Are Likely Broken

These achievements require a like ratio below 20% or above 60% after 50+ swipes. They are listed in `EngagementDomain` but are not checked in `AchievementEventHandler.checkAndUnlock()`, which only runs on match creation (via `SwipeRecorded` with `resultedInMatch=true`). A user could reach 20% like ratio with no matches and never receive the `SELECTIVE` achievement.

**Fix:** Check ratio-based achievements in the metrics path, not only in the match path. Or run `checkAndUnlock()` on every swipe, not just matches.

---

### 3.4 Interest Max Count Not Enforced in UI

`MatchPreferences.Interest.MAX_PER_USER = 10` is defined and enforced in the domain, but `ProfileController.handleEditInterests()` does not disable checkboxes after 10 are selected. Users can select more than 10 interests in the dialog, and the save will then fail server-side with a validation error — with no clear feedback about why.

**Fix:** In the interests dialog, disable unchecked interest toggles once `selectedCount >= MAX_PER_USER`, and show a label like "10 / 10 selected."

---

### 3.5 Message Send Has No Loading State

`ChatController` has no loading overlay or button-disable logic during `sendMessage()`. If the async operation takes time (e.g., storage is slow), the user can click Send multiple times and get duplicate messages. Other controllers (Profile, Matching) handle this correctly with `savingProperty` bindings.

**Fix:** Disable the send button and show an inline indicator while the message async task is in-flight.

---

### 3.6 Profile Note Save Has No Visual Feedback

In both `ChatController` and `MatchingController`, the Save/Delete note buttons are wired but provide no indication of success or failure. The operation completes silently. Other save operations in the app (e.g., profile save) show a status label.

**Fix:** Show a brief "Saved" or "Deleted" status label or toast after the note operation completes.

---

### 3.7 `CandidateFinder` Returns Empty List When User Has No Location

If a user hasn't set their location, `findCandidatesForUser()` immediately returns an empty list. This means the user sees no candidates — not because none exist, but because they haven't entered coordinates. The error is silent from the UI's perspective.

**Fix:** Either show a "Set your location to see matches" prompt in `MatchingViewModel` when the list is empty, or allow discovery by non-location-requiring preferences when no location is set.

---

### 3.8 `UiPresenceDataAccess` Always Reports Offline in MatchesController

`MatchesController` has a comment at the presence-status rendering section: `"FUTURE(presence-tracking)"` and hardcodes "OFFLINE" for all users in the matches list. Meanwhile `ChatViewModel` does have a presence property. This creates an inconsistency: once the chat is opened the presence might update, but the matches list always shows offline.

## 4. Storage Gaps

### 4.1 No Pagination on Candidate Queries

`UserStorage.findCandidates()` returns the full result set; there is no `offset`/`limit` support. For deployments with thousands of active users, this loads the entire filtered candidate pool into memory. The JDBI override adds bounding-box SQL filtering but still returns all results within the box.

**Fix:** Add `int offset, int limit` parameters to `findCandidates()` and update the JDBI implementation with `LIMIT`/`OFFSET`.

---

### 4.2 N+1 Query in Default `countMessagesByConversationIds`

The `CommunicationStorage` interface provides a default implementation that issues one `COUNT(*)` query per conversation ID. This is correctly overridden in `JdbiConnectionStorage` with a `GROUP BY` query, but any future implementation that doesn't override this will silently run N+1 queries.

**Fix:** Remove the default implementation and make the method abstract to force implementors to optimize it.

---

### 4.3 Missing Index for Candidate Geo-Filter

The candidate query filters by `(state, gender, birth_date, lat, lon)` but there is no composite index on those columns. As the user table grows, candidate lookup will do a full scan of the `state = 'ACTIVE'` subset.

**Fix:** Add a composite index: `CREATE INDEX idx_users_discovery ON users (state, gender, birth_date, lat, lon) WHERE deleted_at IS NULL`.

---

### 4.4 Block/Unblock Pagination Missing

`TrustSafetyStorage.getBlockedUserIds(UUID)` returns the full list with no pagination. A user who has blocked many people loads all UUIDs into memory at once.

**Fix:** Add a paginated variant or at minimum a `countBlocked(UUID)` method.

## 5. Event System Gaps

### 5.1 Block and Report Actions Fire No Events

`SocialUseCases.blockUser()` and `reportUser()` both complete successfully and write to storage, but neither fires an event. This means:
- `MetricsEventHandler` cannot update block/report counters reactively
- No notifications are sent to either party
- `GUARDIAN` achievement cannot trigger (as noted above)

---

### 5.2 `MessageSent` Event Path Is Ambiguous

`MessagingUseCases.sendMessage()` calls metrics directly (`activityMetricsService.recordActivity()`) rather than publishing a `MessageSent` event. The defined `AppEvent.MessageSent` event appears to be unused. The event bus is supposed to decouple this kind of cross-cutting concern.

**Fix:** Fire `MessageSent` from the use-case and remove the direct `activityMetricsService` call from the messaging path.

---

### 5.3 No Events for Friend Requests

A friend request received from another user produces no event and no real-time notification. The `FriendRequestAccepted` event exists, but `FriendRequestReceived` does not. Pending requests are only discoverable by polling `SocialUseCases.pendingFriendRequests()`.

**Fix:** Add `AppEvent.FriendRequestReceived` and fire it from `SocialUseCases.requestFriendZone()`.

---

### 5.4 GUI ViewModels Don't Subscribe to Event Bus

There is no evidence that any ViewModel subscribes to `AppEventBus`. If a match is created while the user is on the Dashboard, the `DashboardViewModel` won't refresh its match count until the user triggers a manual refresh. The event bus is wired for metrics and notifications, but not for reactive UI updates.

**Fix:** Add selective event subscriptions in relevant ViewModels — e.g., `DashboardViewModel` subscribes to `MatchCreated` to bump the match counter.

## 7. Quality-of-Life Improvements

These are smaller improvements that would noticeably improve developer or user experience.

### 7.1 No "Unsaved Changes" Warning in ProfileController

`ProfileController` is one of the largest controllers (1,356 lines) and manages a complex form. If a user edits their bio, then navigates away without saving, all changes are lost silently. Other CRUD apps typically warn "You have unsaved changes."

**Fix:** Track a `dirty` flag in `ProfileViewModel` that's set on any field change. Bind it to a prompt in `NavigationService.canNavigateAway()` or similar.

---

### 7.2 Magic Numbers Should Be in `UiConstants`

Several controllers have inline magic numbers:
- `DRAG_THRESHOLD = 150` in `MatchingController`
- `MAX_FLOATING_HEARTS = 20` in `MatchesController`
- Message max length hardcoded in `ChatController`

These values belong in `UiConstants` for consistency and easy adjustment.

---

### 7.3 `RecommendationService.formatDuration()` Breaks Above 24 Hours

The method formats a `Duration` as `HH:mm:ss`. If the daily limit reset is more than 24 hours away, it would show `"25:30:00"` instead of `"1 day 1:30:00"`. This is likely fine in practice (daily resets happen within 24h), but the method should be defensive.

---

### 7.4 Age Timezone Ambiguity

`User` has a deprecated `getAge()` with no timezone argument, plus a preferred overload requiring an explicit `ZoneId`. The deprecated method uses the JVM default timezone. For age-sensitive matching (where an off-by-one near midnight could flip someone in or out of a dealbreaker range), this matters.

**Fix:** Remove the deprecated `getAge()` and ensure all call sites pass an explicit timezone.

---

### 7.5 Polling Intervals Are Hardcoded in ChatViewModel

Conversation list refresh runs every 15 seconds; active conversation refresh every 5 seconds. These are constants in the class, not config values. For real-world use, these should come from `AppConfig` so they can be tuned without code changes.

---

### 7.6 Profile Save Failure Message Is Not Actionable

When `ProfileController.handleSave()` fails, it calls `showSaveFailureStatus()`, but the error message doesn't indicate which field caused the failure. The user has to guess. `ProfileUseCases.saveProfile()` returns a `UseCaseError` with a code; that code should be translated into a field-specific message.

---

### 7.7 Height Field Accepts Negative Numbers

`ProfileController`'s height input validates against a range (`minHeight` to `maxHeight`) but doesn't explicitly check for negative values. A user entering `-170` would pass the range check if `minHeight` is negative.

## 10. Achievement System Disconnect (Second Pass — Critical)

This is the most architecturally significant finding from the second pass.

### Two Parallel Achievement Systems With No Bridge

The codebase has **two completely separate achievement enumerations** that were built independently and never wired together:

**Backend domain model** — `EngagementDomain.Achievement` (11 entries):
```
FIRST_SPARK, SOCIAL_BUTTERFLY, POPULAR, SUPERSTAR, LEGEND,
SELECTIVE, OPEN_MINDED, COMPLETE_PACKAGE, STORYTELLER, LIFESTYLE_GURU, GUARDIAN
```
Used by `DefaultAchievementService`, `StatsViewModel`, `StatsController`. Backed by `analyticsStorage.saveUserAchievement()`. These are the achievements that can actually be unlocked.

**UI popup model** — `MilestonePopupController.AchievementType` (10 entries):
```
FIRST_MATCH, PROFILE_COMPLETE, FIRST_MESSAGE, TEN_MATCHES,
TWENTY_FIVE_MATCHES, FIFTY_MATCHES, FIRST_DATE, ACTIVE_WEEK, SUPER_LIKER, PHOTO_PERFECT
```
Each has an MDI2 icon literal, i18n keys, and an XP value. Carries no connection to the domain model.

**The critical symptom:** `MilestonePopupController.showAchievement(AchievementType type)` is **defined but never called** from any source file. The entire `AchievementType` enum — including its i18n keys, XP values, and icon codes — is dead code.

**What this means in practice:**
- When a user earns `FIRST_SPARK` (domain), the achievement is saved to the database and shown in `StatsController` as a list item with an emoji. No celebratory popup fires.
- The `MilestonePopupController` shows match celebrations (the "IT'S A MATCH!" popup) but its `showAchievement(AchievementType)` overload is orphaned.
- There are 14 i18n keys in `messages.properties` for `ui.achievement.*` (e.g., `ui.achievement.first_match.name`) that are never loaded at runtime.

**Fix options:**
1. Add a mapping from `EngagementDomain.Achievement` → `MilestonePopupController.AchievementType` and call `showAchievement()` from `AchievementEventHandler` after unlock.
2. Or collapse the two systems: remove `AchievementType` and extend `EngagementDomain.Achievement` with the MDI2 icon codes and XP values it currently lacks, then wire `showAchievement(Achievement)` directly.

## 13. Configuration Issues (Second Pass)

### 13.1 `maxPhotos = 2` Is Unusually Restrictive

`ValidationConfig.maxPhotos` defaults to `2`. This is the hard cap on how many photos a user can upload. For reference: Tinder allows 9, Bumble 6, Hinge 6. Two photos makes the app feel demo-like and limits profile quality signals.

**Fix:** Raise default to at least 6 in `AppConfig.ValidationBuilder`.

### 13.2 Standout Scoring Weights Not Validated

`AppConfigValidator.validateAlgorithm()` checks that all 16 algorithm config values are non-negative, but does **not** validate that the 6 standout scoring weights sum to 1.0. By contrast, `validateMatching()` enforces `Math.abs(weightSum - 1.0) > 0.01` for matching weights.

The 6 standout weights in `AlgorithmConfig` (distance, age, interest, lifestyle, completeness, activity) could sum to 0.5 or 2.0 with no error.

**Fix:** Add the same weight-sum validation in `validateAlgorithm()` for standout weights.

### 13.3 Session Timeout Scope Is Misleading

`SafetyConfig.sessionTimeoutMinutes = 5` exists and is used — but only for **swipe/activity sessions** (`SwipeState.Session`) managed by `ActivityMetricsService`. It has no effect on `AppSession` (the login session). A logged-in user is never automatically logged out regardless of inactivity.

The config key name implies authentication-level session control, but it only governs metrics session grouping. This can mislead developers expecting a security-relevant timeout.

**Fix:** Either implement real auto-logout in `AppSession` using this config value, or rename the config key to `swipeSessionTimeoutMinutes` to accurately reflect its scope.

## 14. Design & Maintenance Risks (Second Pass)

### 14.1 `ProfileActivationPolicy` Duplicates `User.isComplete()`

`ProfileActivationPolicy.canActivate()` manually enumerates every required field (name, bio, birthDate, gender, interestedIn, etc.) — the same list that `User.isComplete()` already checks. The code even has a comment acknowledging this:

> *"Mirrors User.isComplete() logic — if isComplete() changes, update this too"*

This is a maintenance trap. If a new required field is added to `User.isComplete()` but the developer forgets to update `ProfileActivationPolicy`, profiles that should fail activation will pass it.

**Fix:** Simplify `ProfileActivationPolicy.canActivate()` to call `user.isComplete()` rather than re-enumerating fields.

### 14.2 Location Validity Check Fails for (0°, 0°)

`ProfileCompletionSupport.java:147` determines whether a user has set their location with:
```java
user.getLat() != 0.0 || user.getLon() != 0.0
```

A user at coordinates (0.0, 0.0) — in the Gulf of Guinea, off the coast of West Africa — would have their location treated as "not set." This causes them to receive 0 points toward profile completion even though they entered a valid location, and they may never see candidates (since `CandidateFinder` returns empty for users with no location).

**Fix:** Use a nullable sentinel value (`null` latitude/longitude) rather than relying on (0, 0) as a missing-location marker.

### 14.3 `PerformanceMonitor` Listed in CLAUDE.md but Does Not Exist

The `CLAUDE.md` architecture section lists `PerformanceMonitor` as a core utility in the `core/` package. The source file does not exist. Any developer following the CLAUDE.md as a guide to the codebase would have an inaccurate mental model.

**Fix:** Either create `PerformanceMonitor.java` with basic timing/counting utilities, or remove it from CLAUDE.md.

### 14.4 Enum Codecs Not Centralized
